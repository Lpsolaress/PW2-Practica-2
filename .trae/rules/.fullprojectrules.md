## Arquitectura obligatoria (NUNCA violar)

El proyecto sigue arquitectura en capas estricta:

```
routers/     → solo HTTP: parsear request, llamar al service, devolver response
services/    → lógica de negocio pura: NO importar Request/Response de FastAPI
repositories/→ ÚNICO punto de acceso a la DB: NO contienen lógica de negocio
models/      → clases ORM SQLAlchemy: NO contienen lógica de negocio
schemas/     → Pydantic: validación de entrada/salida
```

**Reglas de dependencia**:
- Routers → Services → Repositories → Models
- Los servicios NO conocen FastAPI (Request, Response, HTTPException, UploadFile)
- Los repositorios SOLO reciben la sesión SQLAlchemy y primitivas Python
- NUNCA pongas lógica de negocio en routers ni en models

## Contrato de API (NO modificar)

El frontend Svelte consume exactamente estas rutas. Respetar URLs, métodos y estructura JSON:

### Auth
- POST /auth/login       → { token, usuario: {id, username, email, role}, mensaje }
- POST /auth/registro    → { token, usuario: {id, username, email, role}, mensaje }
- GET  /auth/perfil      → { usuario: {...} }
- PUT  /auth/perfil      → { mensaje, usuario: {...} }  (multipart/form-data)
- POST /auth/direcciones → { mensaje, addresses: [...] }
- DELETE /auth/direcciones/{id}
- POST /auth/metodos-pago
- DELETE /auth/metodos-pago/{id}

### Productos (campo `_id` obligatorio, no `id`)
- GET    /productos       → [ {_id, nombre, precio, descripcion, imagen, categoria, plataforma, activo, createdAt, updatedAt}, ... ]
- GET    /productos/{id}
- POST   /productos       → requiere rol administrador, multipart/form-data
- PUT    /productos/{id}  → requiere rol administrador, multipart/form-data
- DELETE /productos/{id}  → requiere rol administrador

### Usuarios (campo `_id` obligatorio)
- GET    /usuarios        → requiere rol administrador
- GET    /usuarios/{id}   → requiere rol administrador
- POST   /usuarios        → requiere rol administrador
- PUT    /usuarios/{id}   → requiere rol administrador
- DELETE /usuarios/{id}   → requiere rol administrador

## Serialización JSON crítica

El frontend usa `producto._id` y `usuario._id` (con guion bajo).
Los modelos ORM usan `id` (sin guion bajo).
**SIEMPRE** usa `serialization_alias="_id"` en los schemas Pydantic de respuesta:

```python
# CORRECTO
class ProductoResponse(BaseModel):
    id: str = Field(serialization_alias="_id")
    model_config = ConfigDict(from_attributes=True, populate_by_name=True)

# INCORRECTO — rompe el frontend
class ProductoResponse(BaseModel):
    id: str  # El frontend espera _id, esto falla
```

Los timestamps deben salir como strings ISO 8601 (`createdAt`, `updatedAt`):
```python
created_at: datetime = Field(serialization_alias="createdAt")
updated_at: datetime = Field(serialization_alias="updatedAt")
```

## Autenticación y autorización

- El token JWT se pasa siempre como `Authorization: Bearer <token>`
- El payload del JWT debe contener `{"sub": user_id}`
- Las rutas de productos (GET) son **públicas** (no requieren token)
- Las rutas de productos (POST/PUT/DELETE) requieren rol `administrador`
- Todas las rutas de `/usuarios/` requieren rol `administrador`
- Las rutas `/auth/perfil` (GET y PUT) requieren cualquier usuario autenticado

Errores de auth esperados:
- 401 → token ausente, inválido o expirado
- 403 → autenticado pero sin permisos de administrador

---

## Manejo de errores

**NUNCA** lanzar `HTTPException` desde services o repositories.
Usa siempre excepciones de dominio de `app.exceptions`:

```python
# CORRECTO (en service)
if not producto:
    raise ProductoNoEncontradoError()

# INCORRECTO
raise HTTPException(status_code=404, detail="Producto no encontrado")
```

El router puede atrapar `HTTPException` si es necesario, pero preferir el handler global.

Formato de error unificado:
```json
{ "error": "mensaje descriptivo" }
```

Para errores de validación Pydantic (422):
```json
{ "error": "Datos de entrada inválidos", "detalles": [...] }
```

## Base de datos

- IDs: siempre UUID v4 como string (`str(uuid.uuid4())`)
- NUNCA usar `Integer` como clave primaria (incompatible con el frontend)
- Timestamps: `created_at` y `updated_at` con `onupdate=datetime.utcnow`
- La sesión de DB se inyecta vía `Depends(get_db)` en los routers, nunca instanciada directamente
- Los repositorios reciben la sesión en el constructor: `def __init__(self, db: Session)`
- Al iniciar la app, llamar a `init_db()` para crear las tablas automáticamente

## Subida de imágenes

- Guardar en la carpeta `/uploads/` en la raíz del proyecto
- Productos: nombre de archivo `{timestamp_ms}{extension}` (ej: `1716200000000.jpg`)
- Fotos de perfil: `profile-{timestamp_ms}{extension}`
- Extensiones permitidas: `.jpg`, `.jpeg`, `.png`, `.webp`
- Devolver la URL como string: `"/uploads/nombre_archivo.jpg"`
- Servir con `StaticFiles` montado en `/uploads`

## Lo que NO hacer

- ❌ No usar `@app.route` (sintaxis Flask), usar `@router.get`, `@router.post`, etc.
- ❌ No acceder a `db` directamente en routers ni services
- ❌ No devolver objetos ORM directamente desde routers (siempre pasar por schema Pydantic)
- ❌ No usar `response_model=dict` — definir siempre un schema explícito
- ❌ No guardar contraseñas en texto plano
- ❌ No hardcodear `SECRET_KEY` en el código — siempre desde `.env`
- ❌ No usar persistencia simulada (listas en memoria, archivos JSON, etc.)
- ❌ No poner toda la lógica en un solo archivo
- ❌ No usar `Optional[X]` de `typing` en Python 3.10+ — usar `X | None`
